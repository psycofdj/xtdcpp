Format specification

  • A configuration has:
    • zero–or-more section
  • A section has:
    • a name
    • zero-or-more subsection
    • zero-or-more property
  • A subsection has:
    • a name
    • zero-or-more property
  • A property has:
    • a key
    • a value

Example:

[myapp]
{
[logging] {
level : info
file : /dev/log
}
[webserver] {
port : 8080
interface : 0.0.0.0
}
}

Running the parser

Two ways of using the Parser class :

  • use default constructor then call one of the following overloads :

    • parse<Iterator>(Iterator start, Iterator end) : parses configuration data from start to end
    • parse(istream&) : parses configuration from given istream
    • parse(const string&) : parses configuration from given file path

    You'll get return status through the result of the parse methods.

  • use one of the following constructor overloads :

    • Parser(istream&, const t_data&)
    • Parser(const string&, const t_data&)

    Theses are equivalent to the first method but you'll have to catch parse_error exception.

Reading keys

Each property of the configuration file is accessible by its key formed by :

  • < section_name > . < subsection_name > . < key_name > : for subsection properties
  • < section_name > . < key_name > : for section properties

You can retrieve the value for a given key by calling one of the following methods :

  • Parser::get(const string&, string&) const : get key as string value
  • Parser::get(const string&, const char*&) const : get key as char* value
  • Parser::get(const string&, bool&) const : get key as boolean value
  • Parser::get<T>(const string&, T&) const : get key lexically casted to T value
  • Parser::get_default<T>(const string&, T&, const T&) const, get key value, use default if error

Alternatively, you can retreive a set of key prefixed by a certain parttern using the function search(const string&) const;

Variables in property values

Property value starts at first non-blank character after colon and ends at the end of the line.

It may contain three types of variable reference.

  • $ENV{name}

    Will be replaced by environment valiable name

  • $PARAM{name}

    Will be replaced by the value of name in the parameter map given at constructor. See Parser(const t_data&).

  • ${key}

    Will be replaced by the value of key in the configuration.

    Note that ${key} may contain references to other variables. Variable resolution will go until no more references are found or a cycle reference is detected.

Example : Given an environment variable password equals to secret, the two following configuration file are equivalent :

  • [myapp]
    {
    [mysql] {
    host : localhost
    port : 3306
    user : demo
    pass : prefix_$ENV{password}
    }
    [service1] {
    odbc_string : mysql://${myapp.mysql.user}:${myapp.mysql.pass}@${myapp.mysql.host}:${myapp.mysql.port}/service1
    }
    [service2] {
    odbc_string : mysql://${myapp.mysql.user}:${myapp.mysql.pass}@${myapp.mysql.host}:${myapp.mysql.port}/service2
    }
    }
  • [myapp]
    {
    [mysql] {
    host : localhost
    port : 3306
    user : demo
    pass : secret
    }
    [service1] {
    odbc_string : mysql://demo:prefix_secret@localhost:3306/service1
    }
    [service2] {
    odbc_string : mysql://demo:prefix_secret@localhost:3306/service2
    }
    }
